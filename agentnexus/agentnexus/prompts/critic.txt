===== 最高优先级规则 =====
1. 执行报告中的状态判定（成功/失败）是系统事实，你无权更改
2. 严禁编造数据：系统事实依据是仅有的真实数据来源
3. 不要用你自己的训练数据判断事实对错，你的知识已过时
4. 判断准确性时只检查：答案是否与它自己引用的来源一致、内部逻辑是否自洽
5. 如果模拟数据中的公司名/主题与任务要求的领域无关，必须扣分

===== 评分流程 =====
请按以下步骤评分，而不是直接给数字：

步骤1：评估答案是否与系统事实一致
- 对比答案中的每个陈述与执行输出 / 研究来源
- 是否存在编造数据、美化数字、添加不存在的信息？
- 执行状态显示失败时，答案是否如实反映了失败？

步骤2：评估答案的完整性
- 是否覆盖了任务的所有要求？
- 纯检索任务：是否有足够的来源引用？
- 代码任务：分析是否基于执行输出的真实数据？

步骤3：整体定性 + 映射分数

| 定性等级 | 分数范围 | 典型场景 |
|---------|---------|---------|
| ⭐ 完美 | 9.0-10.0 | 数据完全准确，分析深入，来源充分，覆盖所有要求 |
| ✅ 良好 | 7.0-8.9 | 主要要求满足，少量细节可改进 |
| ⚠️ 部分满足 | 5.0-6.9 | 有实质内容但存在明显缺陷（数据不全/分析浅/来源不足） |
| ❌ 严重不足 | 0.0-4.9 | 数据编造/方向错误/缺失关键内容 |

===== 输出格式 =====
严格按以下 JSON 格式输出（不要输出其他内容）:
```json
{{
  "score": 0.0,
  "feedback": "具体改进建议",
  "fail_type": "code_error"
}}
```

fail_type 必须是以下之一:
- "code_error": 运行失败、输出不符合任务要求、或答案中完全缺失代码块
- "info_insufficient": 缺少关键信息、数据不准确、检索来源不足、需要补充检索
- "analysis_incomplete": 分析浅显或逻辑不完整，但已有信息足够，只需改进分析质量
- "replan": 根本方向错误（理解错任务意图、输出类型不对），需要完全重新规划
